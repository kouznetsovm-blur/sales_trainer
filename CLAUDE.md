# Zori — голосовой тренер для продавцов

## Описание проекта
Веб-приложение для голосового тренинга продавцов. Двусторонний голосовой диалог в реальном времени без нажатия кнопок в процессе общения.

## Документация
Вся документация находится в `docs/`:
- `architecture-spec.md` — архитектура, структура проекта, схема взаимодействия, БД
- `business-requirements.md` — бизнес-требования (визуальный интерфейс)
- `voice-module-spec.md` — требования к голосовому модулю

**Перед любыми изменениями — читать документацию.**

## Рабочий процесс
1. Изменения вносятся локально
2. Локальные изменения пушатся на VPS (`root@161.35.68.238:/var/www/sales_trainer`)
3. С VPS изменения пушатся в GitHub

**Никогда не пушить напрямую с мака в GitHub.**

**Мак, VPS и GitHub никогда не должны расходиться.** После каждого изменения:
- Локальные файлы → VPS: `./deploy.sh`
- VPS → GitHub: `git add . && git commit -m "..." && git push`

Не оставлять изменения только на одном из трёх узлов.

## Технический стек
- **Фронтенд:** React + Vite
- **Бэкенд:** Node.js + Express
- **БД:** SQLite (better-sqlite3)
- **Голос:** OpenAI Realtime API (WebRTC), модель `gpt-realtime-mini`

## Структура проекта
```
004_sales_trainer/
├── client/        ← React + Vite
├── server/        ← Node.js + Express
└── docs/          ← ТЗ
```

## Важные правила
- `.env` не коммитить (хранит `OPENAI_API_KEY`)
- API ключ OpenAI хранится только на сервере, фронтенд его не видит

## Процесс разработки (строго соблюдать)
1. Сначала **бизнес-требования** (`docs/business-requirements.md`) — что и зачем строим
2. Из бизнес-требований рождается **техническое задание** (`docs/`) — как строим
3. Только после согласования ТЗ — **код**

**Все три части никогда не должны расходиться.**

При любом изменении — обязательно проверить остальные два компонента:
- Изменились **бизнес-требования** → проверить, нужно ли обновить ТЗ и код
- Изменилось **ТЗ** → проверить соответствие бизнес-требованиям и обновить код
- Изменился **код** → проверить соответствие ТЗ и бизнес-требованиям, при необходимости обновить документы

Если расхождение обнаружено — сначала устранить его, потом продолжать работу.
